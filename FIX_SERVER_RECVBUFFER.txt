FIX MINIMO A src/Server.cpp — accesso a _recvBuffer
=====================================================
(applicato da Elia durante il merge di origin/main nel branch elmondo)

IL PROBLEMA
-----------
Server.cpp non compilava:

    src/Server.cpp:25: error: 'std::string Client::_recvBuffer' is private within this context
    (stesso errore alle righe 34, 37 e 100)

In Client.hpp il membro _recvBuffer sta nella sezione `private:` della classe
Client. Server non è una classe friend, quindi non può accedere direttamente
a _recvBuffer. Il codice però lo faceva in 4 punti.

COM'ERA PRIMA
-------------
    void	Server::readBuffer( int fd ) {
        int bytes;
        char	tmp[512];
        while (_clients[fd]._recvBuffer.find("\r\n") != std::string::npos) {
            bytes = recv(fd, tmp, 512, 0);
            if (bytes < 0) {
                throw NetworkError();
            }
            else if (bytes == 0) {
                break ;
            }
            else {
                _clients[fd]._recvBuffer.append(tmp, bytes);
            }
        }
        std::cout << _clients[fd]._recvBuffer << std::endl;
    }

e nella run(), riga 100:

    IrcMessage msg = parseMessage(_clients[it->fd]._recvBuffer);

COSA HO CAMBIATO
----------------
Solo questo, in 4 righe: `_clients[fd]._recvBuffer` --> `_clients[fd].getRecvBuffer()`

getRecvBuffer() esiste già in Client (l'avete scritto voi) e restituisce
`std::string&`, cioè un RIFERIMENTO al buffer vero, non una copia:

    std::string &Client::getRecvBuffer()
    {
        return (_recvBuffer);
    }

Quindi il comportamento è IDENTICO a prima: .find() legge il buffer vero,
.append() scrive nel buffer vero. Cambia solo che ci si passa dal getter
pubblico invece che dal campo privato. Zero cambi di logica.

PERCHÉ COSÌ E NON friend/public
-------------------------------
- rendere _recvBuffer public romperebbe l'incapsulamento per tutti
- `friend class Server` funzionerebbe ma lega Client a Server senza motivo
- il getter pubblico esisteva già ed era fatto apposta per questo

NOTA A PARTE (da guardare, non l'ho toccata)
--------------------------------------------
La condizione del while in readBuffer sembra invertita:

    while (_clients[fd].getRecvBuffer().find("\r\n") != std::string::npos)

entra nel loop solo se il buffer GIÀ contiene "\r\n" — ma il buffer si
riempie proprio dentro il loop con recv(). Alla prima chiamata il buffer è
vuoto, find() torna npos, il while non entra mai e non si legge niente dal
socket. Probabilmente la condizione voleva essere `== std::string::npos`
("continua a leggere FINCHÉ non arriva un \r\n completo").
Attenzione però: con == npos e un client lento il loop bloccherebbe il
server (recv bloccante in loop). Da ragionare insieme.

Inoltre dopo il dispatch il buffer non viene mai svuotato/consumato riga
per riga, quindi i comandi già processati restano dentro e verrebbero
riparsati al giro successivo.
