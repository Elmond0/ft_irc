1. COS'E' IRC
--------------
IRC (Internet Relay Chat) e' un protocollo di messaggistica testuale in tempo
reale definito nell'RFC 1459 (1993) e aggiornato negli RFC 2810-2813 (2000).
E' uno dei protocolli di rete piu' longevi ancora in uso, e costituisce un
esempio eccellente di architettura client-server basata su testo ASCII.

Un server IRC e' un processo unico in ascolto su una porta TCP (solitamente
la 6667). I client si connettono tramite socket TCP e comunicano con il server
inviando linee di testo terminate da CRLF (\r\n). Il server interpreta queste
linee come "messaggi IRC" composti da:

   [':' prefix SPACE] command [params] [':' trailing] CRLF

Esempio reale di messaggio IRC:
   PRIVMSG #42firenze :ciao a tutti!\r\n
   :nick!user@host KICK #canale vittima :motivo\r\n

Il server mantiene in memoria due strutture fondamentali:
  - Client: ogni connessione TCP aperta, identificata dal proprio file descriptor
  - Channel: ogni canale aperto (inizia con '#'), con la lista dei membri

2. CICLO DI VITA DI UN CLIENT IRC
-----------------------------------
Un client prima di poter usare qualsiasi comando deve REGISTRARSI. La
registrazione avviene obbligatoriamente in questo ordine:

  1) PASS  <password>     — verifica la password del server
  2) NICK  <nickname>     — imposta il proprio soprannome
  3) USER  <user> 0 * :<realname>  — imposta username e nome reale

Solo dopo che tutti e tre sono andati a buon fine il server invia i messaggi
di benvenuto (RPL_WELCOME 001 ... 004) e il client e' "registrato": da quel
momento puo' usare JOIN, PRIVMSG, KICK, ecc.

3. REPLY NUMERICHE
-------------------
Il protocollo IRC usa codici numerici a tre cifre per le risposte del server,
esattamente come HTTP usa 200, 404, 500. Esempi usati in questo progetto:

  001  RPL_WELCOME         — benvenuto, registrazione completata
  331  RPL_NOTOPIC         — nessun topic impostato nel canale
  332  RPL_TOPIC           — topic attuale del canale
  341  RPL_INVITING        — conferma di INVITE inviato
  353  RPL_NAMREPLY        — lista dei nick nel canale (con @ per gli op)
  366  RPL_ENDOFNAMES      — fine della lista nomi
  401  ERR_NOSUCHNICK      — nick o canale inesistente
  403  ERR_NOSUCHCHANNEL   — canale inesistente
  404  ERR_CANNOTSENDTOCHAN— non puoi inviare messaggi a quel canale
  411  ERR_NORECIPIENT     — nessun destinatario per PRIVMSG
  412  ERR_NOTEXTTOSEND    — nessun testo in PRIVMSG
  421  ERR_UNKNOWNCOMMAND  — comando sconosciuto
  431  ERR_NONICKNAMEGIVEN — nessun nickname fornito
  432  ERR_ERRONEUSNICKNAME— nickname non valido (caratteri non ammessi)
  433  ERR_NICKNAMEINUSE   — nickname gia' in uso da un altro client
  441  ERR_USERNOTINCHANNEL— il target non e' nel canale
  442  ERR_NOTONCHANNEL    — il client non e' nel canale
  443  ERR_USERONCHANNEL   — il target e' gia' nel canale (INVITE)
  451  ERR_NOTREGISTERED   — non ancora registrato
  461  ERR_NEEDMOREPARAMS  — parametri mancanti
  462  ERR_ALREADYREGISTRED— tentativo di ri-registrarsi
  464  ERR_PASSWDMISMATCH  — password errata
  471  ERR_CHANNELISFULL   — canale pieno (+l)
  472  ERR_UNKNOWNMODE     — carattere di modalita' non riconosciuto
  473  ERR_INVITEONLYCHAN  — canale solo su invito (+i)
  475  ERR_BADCHANNELKEY   — password canale errata (+k)
  482  ERR_CHANOPRIVSNEEDED— operazione richiede privilegi di operatore

4. MODELLO I/O: poll() E I/O MULTIPLEXING
-------------------------------------------
Il cuore del server si trova in Server::run() (src/Server.cpp).

Il problema fondamentale di un server multi-client e' il seguente: come si
puo' attendere dati da centinaia di connessioni contemporaneamente senza
creare un thread per ciascuna?

La soluzione classica in C/C++ e' l'I/O multiplexing con poll(). La funzione
poll() riceve un array di strutture pollfd (file descriptor + eventi attesi)
e si blocca finche' almeno uno di essi diventa pronto per la lettura o la
scrittura. E' il modello "reattivo": il server non fa nulla finche' non
arriva qualcosa.

  struct pollfd {
      int   fd;       // il socket da monitorare
      short events;   // cosa vogliamo (POLLIN, POLLOUT)
      short revents;  // cosa e' successo (riempito da poll())
  };

Nel nostro server:
  - pfds[0] e' sempre il socket di ascolto (listen socket)
  - pfds[1..n] sono i socket dei client connessi

Quando pfds[0] ha POLLIN significa che un nuovo client vuole connettersi:
si chiama accept() per creare un nuovo socket e si aggiunge a pfds.

Quando pfds[i] ha POLLIN arrivano dati da un client esistente: si chiama
recv() per leggere i dati nel buffer.

Questa scelta e' IMPOSTA dal progetto 42 (niente thread, niente fork,
solo poll/select/epoll). Insegna la programmazione asincrona non bloccante.

5. STRUTTURE DATI PRINCIPALI
------------------------------
  std::map<int, Client>        _clients   — mappa fd -> Client
  std::map<std::string, Channel> _channels  — mappa nome -> Channel

La scelta di std::map (albero rosso-nero, O(log n) su tutte le operazioni)
e' deliberata: garantisce ordinamento, accesso per chiave e iterazione
predicibile. Una hash_map avrebbe O(1) medio ma in C++98 non e' standard
(unordered_map arriva in C++11).

Il Client e' identificato dal proprio file descriptor (intero), che il kernel
garantisce essere unico per ogni connessione aperta. Il canale e' identificato
dal nome (stringa), che per RFC inizia con '#'.

6. PARSING: IrcMessage E parseMessage()
-----------------------------------------
File: src/Parser.cpp, inc/IrcMessage.hpp, inc/Parser.hpp

Il primo passo e' convertire la stringa grezza ricevuta via TCP in una
struttura dati semantica. La struct IrcMessage modella esattamente il
formato definito nell'RFC 1459:

  struct IrcMessage {
      std::string              prefix;   // mittente (es. "nick!user@host")
      std::string              command;  // es. "JOIN", "PRIVMSG"
      std::vector<std::string> params;   // parametri posizionali
      std::string              trailing; // parametro dopo ':' (puo' contenere spazi)
  };

La distinzione tra params e trailing e' cruciale: i parametri normali sono
separati da spazi e non possono contenerli; il trailing (introdotto da ':')
puo' contenere qualsiasi carattere inclusi gli spazi. Questo e' il motivo
per cui ":ciao a tutti" diventa un singolo parametro trailing.

La funzione parseMessage() implementa uno stato macchina lineare:
  1. Rimuove il CRLF finale
  2. Se la riga inizia con ':', estrae il prefix fino al primo spazio
  3. Estrae il command come primo token e lo converte in maiuscolo
  4. Raccoglie i parametri successivi uno per uno
  5. Se incontra ':', tutto il resto diventa il trailing

La conversione in maiuscolo tramite toUpper() e' fondamentale per rendere
il comando case-insensitive ("join", "JOIN", "Join" sono tutti validi).

Nota tecnica: il cast `static_cast<unsigned char>` prima di toupper() e'
necessario in C++98 per evitare undefined behavior su sistemi dove char e'
signed e il carattere ha valore > 127 (caratteri estesi).

7. DISPATCHER: TABELLA DI FUNZIONI PUNTATORI A METODI
-------------------------------------------------------
File: src/dispatch.cpp, inc/dispatch.hpp

Il Dispatcher risolve un classico problema: dato un comando come stringa
("JOIN", "KICK"...), come chiamare la funzione corrispondente in modo
efficiente ed estensibile?

La soluzione naive sarebbe una catena di if-else:
  if (cmd == "JOIN") doJoin(); else if (cmd == "KICK") doKick(); ...

Questo e' lento (O(n) confronti) e non estensibile. La soluzione elegante
e' usare una tabella di dispatch: una std::map che associa la stringa del
comando a un puntatore a funzione (o, nel nostro caso, un puntatore a metodo).

  typedef void (Command::*CommandFn)(Client&, const IrcMessage&);
  std::map<std::string, CommandFn> _handlers;

Questa typedef definisce un "puntatore a metodo non-statico della classe
Command che prende Client& e const IrcMessage& e restituisce void".

In initHandlers() la tabella viene popolata una volta sola:
  _handlers["JOIN"]    = &Command::JOIN;
  _handlers["PRIVMSG"] = &Command::PRIVMSG;
  ...

La chiamata avviene tramite la sintassi:
  (cmd.*(it->second))(client, msg);

Questo e' un PATTERN FONDAMENTALE del C++: permette di selezionare a runtime
quale metodo di un oggetto eseguire, senza conoscere a compile-time quale
specifico metodo sia. E' concettualmente simile a una virtual table (vtable)
ma controllata esplicitamente dal programmatore.

Complessita' di dispatch: O(log n) per la ricerca in std::map.

Il Dispatcher gestisce anche la logica di pre-registrazione: alcuni comandi
(PASS, NICK, USER, PING, QUIT) sono ammessi prima che il client sia registrato;
tutti gli altri vengono bloccati con ERR_NOTREGISTERED (451). Questa lista
e' memorizzata in un std::set<std::string> _preReg per ricerca O(log n).

8. GESTIONE DEGLI ERRORI: ECCEZIONI ANNIDATE
----------------------------------------------
File: inc/Commands.hpp, src/cmd/Command.cpp

Il progetto usa eccezioni C++ per propagare gli errori dai comandi fino
al Dispatcher. Questo e' il pattern "throw & catch at boundary":
  - I comandi lanciano eccezioni (NumericError, ecc.)
  - Il Dispatcher le cattura e le converte in reply numeriche IRC

Command::NumericError e' una classe annidata (inner class) di Command.
L'annidamento e' usato per:
  1. Raggruppare logicamente l'eccezione con la classe che la usa
  2. Evitare conflitti di naming nel namespace globale
  3. Rendere il codice auto-documentante: chi legge sa che NumericError
     e' specifica del sistema di comandi IRC

NumericError porta un codice numerico (int _code) e un testo (_text).
Nel catch del Dispatcher, questi vengono formattati come reply IRC:
  ":ircserv 461 nick PASS :Not enough parameters\r\n"

Le eccezioni NotRegisteredException e UnknownCommandException sono classi
annidate di Dispatcher, per lo stesso principio di raggruppamento.

La scelta delle eccezioni (invece di return codes o output parameters) e'
motivata dalla profondita' della call stack: un helper privato come
applyOneMode() puo' segnalare errori che risalgono fino al Dispatcher
senza dover passare error codes attraverso ogni livello intermedio.

9. REGISTRAZIONE: PASS, NICK, USER
-------------------------------------

9.1 PASS (src/cmd/PASS.cpp)
----------------------------
Sintassi: PASS <password>
RFC: 1459 sezione 4.1.1

PASS verifica che la password fornita corrisponda a quella con cui e' stato
avviato il server. E' il primo passo della registrazione.

Logica:
  - Se il client e' gia' registrato: ERR_ALREADYREGISTRED (462)
  - Se mancano parametri: ERR_NEEDMOREPARAMS (461)
  - Se la password non corrisponde: ERR_PASSWDMISMATCH (464)
  - Altrimenti: client.setPassOk(true)

Nota: PASS da solo non completa la registrazione. Il client deve poi inviare
NICK e USER. Solo quando tutti e tre i flag (_passOk, _nickOk, _userOk) sono
true, il metodo isRegistered() della classe Client restituisce true.

Questo "flag di stato" distribuito in tre booleani riflette la natura
incrementale della handshake IRC: i tre messaggi possono arrivare in qualsiasi
ordine (anche se PASS deve essere inviato prima dei comandi autenticati).

9.2 NICK (src/cmd/NICK.cpp)
----------------------------
Sintassi: NICK <nickname>
RFC: 1459 sezione 4.1.2

NICK imposta o cambia il nickname del client. E' il piu' complesso dei tre
comandi di registrazione perche' deve gestire il cambio nick a runtime.

Validazione del nickname (isValidNick()):
  - Lunghezza: 1-9 caratteri (limite storico IRC)
  - Primo carattere: lettera o carattere speciale ([]\`^{}|_-)
  - Caratteri successivi: lettere, cifre, o speciali
  - La stringa speciale inclusa "_-[]\\`^{}|" e' conforme all'RFC 1459

Controllo unicita': il codice itera su tutti i client connessi per verificare
che nessun altro stia gia' usando quel nickname. Complessita' O(n) dove n e'
il numero di client. In un server reale con migliaia di client si userebbe
una mappa inversa nick->fd, ma per questo progetto e' accettabile.

Notifica del cambio nick:
Quando un client GIA' REGISTRATO cambia nick, il server deve notificare
non solo il client stesso ma anche tutti gli altri client che condividono
almeno un canale con esso. Il codice usa std::set<int> notified per
evitare di inviare la stessa notifica piu' volte allo stesso destinatario
(un client potrebbe condividere piu' canali con il richiedente).

Il messaggio inviato ha il VECCHIO prefisso:
  ":vecchionick!user@host NICK :nuovonick\r\n"
Questo e' fondamentale: i client IRC usano il prefisso per aggiornare le
proprie liste locali.

9.3 USER (src/cmd/USER.cpp)
----------------------------
Sintassi: USER <username> 0 * :<realname>
RFC: 1459 sezione 4.1.3

USER imposta username e nome reale. Il secondo e terzo parametro ('0' e '*')
sono campi obsoleti (mode e hostname) che i client moderni ignorano; il
realname e' nel campo trailing (dopo ':') perche' puo' contenere spazi.

Dopo USER (se NICK e' gia' stato ricevuto), isRegistered() diventa true e
viene chiamata sendWelcome() che invia i messaggi numerici 001-004.

10. GESTIONE DEI CANALI: JOIN, PART, QUIT
-------------------------------------------

10.1 JOIN (src/cmd/JOIN.cpp)
------------------------------
Sintassi: JOIN <#canale>[,<#canale2>...] [<key>[,<key2>...]]
          JOIN 0  (esce da tutti i canali)
RFC: 1459 sezione 4.2.1

JOIN e' il comando piu' articolato del gruppo di gestione canali.

La funzione pubblica JOIN() delega tutto il lavoro a joinOne() per gestire
il caso di JOIN multipli (separati da virgola). La separazione e' gestita
da splitComma() in utils.cpp che implementa uno split senza librerie esterne.

La funzione joinOne() implementa la macchina a stati di accesso al canale:

  1. Validazione nome: deve iniziare con '#', non avere spazi o virgole
  2. Creazione automatica: se il canale non esiste viene creato al volo
     con channels.insert(std::make_pair(name, Channel(name)))
  3. Controllo +i (invite-only): se attivo, il client deve essere nella
     lista degli invitati del canale
  4. Controllo +k (key): se la password del canale e' impostata, deve
     corrispondere
  5. Controllo +l (limit): se il limite utenti e' impostato e raggiunto,
     il join viene rifiutato
  6. Accesso concesso: il client viene aggiunto al canale, rimosso dalla
     lista invitati, e se e' il fondatore (isNew == true) diventa operatore

La notifica JOIN viene inviata A TUTTI i membri del canale (incluso chi
ha appena fatto JOIN) tramite broadcastToChannel() con exceptFd = -1.
Questo e' il comportamento standard: il client deve ricevere il proprio
JOIN per confermare l'ingresso.

Poi viene inviato il topic (331 o 332) e la lista nomi (353 + 366).

JOIN 0: il caso speciale "JOIN 0" manda il client fuori da TUTTI i canali.
E' implementato da partAll() che itera su tutti i canali e chiama PART
logicamente. Attenzione all'iterazione: il codice usa il pattern classico
"pre-incremento del current iterator" per poter cancellare l'elemento
durante il ciclo:
  std::map<...>::iterator current = it++;
  // usa current, eventualmente erase(current)
  // it e' gia' avanzato, quindi erase(current) non lo invalida

10.2 PART (src/cmd/PART.cpp)
------------------------------
Sintassi: PART <#canale>[,<#canale2>...] [:<motivo>]
RFC: 1459 sezione 4.2.2

PART e' il complemento simmetrico di JOIN. Il client lascia uno o piu' canali.

La notifica di PART viene inviata a TUTTI i membri (incluso chi esce)
prima di rimuoverlo dalla lista: questo e' necessario per la coerenza del
protocollo, altrimenti chi lascia il canale non riceverebbe la conferma.

Se il canale rimane vuoto dopo il PART, viene eliminato dalla mappa:
  if (chan->isEmpty()) _server.getChannels().erase(name);
Non c'e' la concetto di canale "persistente" in questo server: i canali
esistono solo finche' hanno almeno un membro.

10.3 QUIT (src/cmd/QUIT.cpp)
------------------------------
Sintassi: QUIT [:<motivo>]
RFC: 1459 sezione 4.1.6

QUIT e' la disconnessione volontaria. E' piu' complessa di PART perche' deve:

  1. Notificare tutti i membri di TUTTI i canali in cui si trova il client
  2. Usare std::set<int> notified per non inviare lo stesso messaggio due
     volte (il client potrebbe condividere piu' canali con lo stesso utente)
  3. Rimuovere il client da ogni canale, eliminando i canali vuoti
  4. Inviare "ERROR :Closing Link\r\n" al client uscente
  5. Chiudere il socket con close(fd)
  6. Rimuovere il client dalla mappa _clients

L'ordine delle operazioni e' critico: close(fd) deve avvenire DOPO l'invio
di ERROR e DOPO che fd e' stato salvato in una variabile locale, perche'
_clients.erase(fd) potrebbe distruggere l'oggetto Client che contiene fd.

11. MESSAGGISTICA: PRIVMSG
-----------------------------
Sintassi: PRIVMSG <target> :<testo>
RFC: 1459 sezione 4.4.1

PRIVMSG supporta due tipi di destinatario:

a) Canale (target inizia con '#'):
   - Il canale deve esistere (ERR_NOSUCHCHANNEL -> ma usa 401)
   - Il mittente deve essere nel canale (ERR_CANNOTSENDTOCHAN 404)
   - Il messaggio viene broadcastato a tutti ECCETTO il mittente:
     broadcastToChannel(_server, *chan, line, client.getFd())
     Il parametro exceptFd = client.getFd() esclude il mittente.

b) Utente (target e' un nickname):
   - Trova il client tramite findClientByNick() (ricerca O(n))
   - Invia direttamente al suo socket

La formattazione del messaggio usa userPrefix(client) che restituisce
":nick!user@host", consentendo ai client di vedere chi ha inviato il messaggio.

12. GESTIONE DEGLI OPERATORI: KICK, INVITE
--------------------------------------------

12.1 KICK (src/cmd/KICK.cpp)
------------------------------
Sintassi: KICK <#canale> <nickname> [:<motivo>]
RFC: 1459 sezione 4.2.8

KICK espelle un utente da un canale. Richiede privilegi di operatore.

Catena di validazione:
  1. Controllo parametri (almeno 2)
  2. Il canale deve esistere
  3. Il richiedente deve essere nel canale
  4. Il richiedente deve essere operatore del canale
  5. Il target deve essere nel canale

Se il motivo non e' specificato (trailing vuoto), per convenzione si usa
il nickname di chi ha eseguito il KICK.

La notifica viene inviata a TUTTI i membri (incluso il bersaglio) PRIMA
di rimuoverlo: stessa logica di PART.

12.2 INVITE (src/cmd/INVITE.cpp)
----------------------------------
Sintassi: INVITE <nickname> <#canale>
RFC: 1459 sezione 4.2.7

INVITE aggiunge un utente alla lista degli invitati di un canale +i.

Catena di validazione:
  1. Almeno 2 parametri
  2. Il target deve essere un client connesso
  3. Il canale deve esistere
  4. Il richiedente deve essere nel canale
  5. Se il canale e' +i, solo un operatore puo' invitare
  6. Il target non deve essere gia' nel canale

Se tutto e' valido:
  - chan->addInvited(target) aggiunge alla lista degli invitati
  - Il richiedente riceve RPL_INVITING (341)
  - Il target riceve il messaggio INVITE sul suo socket

La lista degli invitati viene pulita automaticamente quando il target
esegue JOIN (chan.removeInvited(&client) in joinOne()).

13. TOPIC (src/cmd/TOPIC.cpp)
------------------------------
Sintassi: TOPIC <#canale> [:<nuovo topic>]
RFC: 1459 sezione 4.2.4

TOPIC e' bifunzionale: senza argomento legge il topic, con argomento lo imposta.

Lettura:
  Il client e' nel canale ma non ha fornito un nuovo topic:
  viene inviato 331 (no topic) o 332 (topic attuale).

Modifica:
  Se il canale ha la modalita' +t (topic restricted), solo gli operatori
  possono cambiare il topic. La notifica del cambio va a tutti i membri
  del canale (incluso chi ha eseguito il comando).

Il valore del topic viene cercato prima in msg.trailing (forma canonica
":nuovo topic con spazi"), poi in msg.params[1] (forma alternativa senza
spazi). Questo rende il comando compatibile con diversi client IRC.

14. MODE (src/cmd/MODE.cpp)
-----------------------------
Sintassi: MODE <#canale> [<flags> [<argomenti>...]]
RFC: 1459 sezione 4.2.3

MODE e' il comando piu' complesso. Gestisce le "modalita'" di un canale,
ovvero flags che modificano il comportamento del canale.

Modalita' implementate:
  +i / -i   Invite-only: solo chi e' stato invitato puo' fare JOIN
  +t / -t   Topic restricted: solo operatori possono cambiare il topic
  +k <key>  / -k   Imposta/rimuove la password del canale
  +o <nick> / -o   Promuove/degrada un utente a operatore del canale
  +l <n>    / -l   Imposta/rimuove il limite massimo di utenti

La gestione delle modalita' e' implementata in tre funzioni:

applyModes() — coordinatore:
  Itera sui caratteri del flag string (es. "+it-k"). Mantiene il segno
  corrente (adding = true/false) e accumula i flag applicati per costruire
  la stringa di risposta da broadcastare.
  Usa il pattern "lastSign" per comprimere "+i+t" in "+it" nella risposta.

applyOneMode() — applicazione di un singolo flag:
  Usa uno switch su char per gestire ogni modalita'. Questo e' piu' efficiente
  e leggibile di una catena if-else. Per le modalita' che richiedono argomenti
  (k, o, l), chiama nextArg() per consumare il parametro successivo.

nextArg() — consumo degli argomenti:
  E' un metodo statico che avanza un indice sulla lista params. Questo pattern
  di "cursore" e' fondamentale: i parametri di MODE si consumano
  sequenzialmente come in un parser a discesa ricorsiva.

Senza argomento (solo MODE <#canale>), viene chiamata sendChannelModes()
che invia RPL_CHANNELMODEIS (324) con la stringa delle modalita' attive.

15. PING (src/cmd/PING.cpp)
-----------------------------
Sintassi: PING <token>
Risposta: PONG <server> :<token>
RFC: 1459 sezione 4.6.2

PING e' il keepalive del protocollo IRC. I client inviano periodicamente PING
al server per verificare che la connessione sia ancora attiva; il server
risponde con PONG ripetendo il token ricevuto.

Il token viene cercato prima in msg.trailing poi in msg.params[0], per
compatibilita' con client che usano le due forme.

PING e' nell'elenco _preReg: e' ammesso anche prima della registrazione.
Questo e' necessario perche' alcuni client inviano PING durante la handshake.

16. LA CLASSE Channel (src/Channel.cpp, inc/Channel.hpp)
----------------------------------------------------------
Channel e' una classe a responsabilita' singola: gestisce lo stato di un
canale IRC. Non conosce socket, non conosce parsing, non invia nulla
direttamente.

Struttura interna:
  std::vector<Client*> _clients    — lista dei membri (puntatori)
  std::vector<Client*> _operators  — lista degli operatori (sottoinsieme di _clients)
  std::vector<Client*> _invited    — lista degli invitati (potenziali membri)

Il puntatore Client* e' la scelta giusta: i Client vivono nella mappa
_clients del Server; Channel non possiede i Client, li "osserva". Questo
evita la copia degli oggetti Client (che sono grandi: hanno buffer di I/O)
e garantisce che tutti puntino allo stesso oggetto in memoria.

I metodi contains() e remove() sono static private: operano su un
std::vector<Client*> generico e sono utility interne riutilizzate.
L'usso di static qui significa che non accedono a dati di istanza, solo
al vettore passato come parametro.

getModeString() restituisce la stringa corrente delle modalita' (es. "+itk"):
  itera sui flag booleani e aggiunge la lettera corrispondente.

Il metodo broadcast() implementa l'invio a tutti i membri con esclusione
facoltativa, utile per PRIVMSG (non inviare al mittente) e JOIN (inviare
a tutti incluso il nuovo membro).

explicit Channel(const std::string& name):
  Il qualificatore explicit impedisce le conversioni implicite da std::string
  a Channel. Questo e' C++ buona pratica: un costruttore con un solo parametro
  non-void dovrebbe quasi sempre essere explicit per evitare conversioni
  accidentali che nascondono bug.

17. LA CLASSE Client (src/Client.cpp, inc/Client.hpp)
-------------------------------------------------------
Client modella una singola connessione TCP. Contiene:
  - Il file descriptor del socket (_sock_fd)
  - L'indirizzo IP/porta del client (sockaddr_in _addr)
  - I dati di registrazione (nick, user, realname, hostname)
  - I flag di stato (_passOk, _nickOk, _userOk, _registered)
  - Due buffer di I/O: _sendBuffer e _recvBuffer

Il sistema a buffer e' fondamentale per I/O asincrono:
  _recvBuffer: accumula i dati ricevuti (recv() puo' arrivare in pezzi)
  _sendBuffer: accoda messaggi da inviare (send() potrebbe non inviare tutto)

getPrefix() restituisce "nick!user@host", il formato standard IRC del prefisso.

isRegistered() restituisce true solo se tutti e tre i flag sono settati.
Questo metodo e' il punto di controllo centralizzato della registrazione,
evitando di duplicare la logica (flag1 && flag2 && flag3) in piu' posti.

25. PERCHE' C++98 E NON C++11/14/17?
---------------------------------------
Il progetto 42 impone C++98 (flag -std=c++98 nel Makefile). Questo e' una
scelta pedagogica: C++98 non ha lambda, auto, range-for, smart pointer,
unordered_map. Lo studente deve conoscere i costrutti fondamentali senza le
"scorciatoie" moderne.

In C++11 il dispatcher sarebbe piu' compatto con std::function e lambda:
  _handlers["JOIN"] = [](Command& c, Client& cl, const IrcMessage& m) { c.JOIN(cl,m); };
In C++98 usiamo puntatori a metodo: piu' verbosi ma fondamentalmente identici.

26. PERCHE' UN SOLO THREAD?
-----------------------------
Poll() con un singolo thread e' sufficiente per un server IRC a basso
traffico. I vantaggi:
  - Nessuna sincronizzazione (mutex, race condition)
  - Codice deterministico e debuggabile
  - Un singolo evento alla volta: nessuna concorrenza sulle strutture dati

Uno svantaggio: un comando che impiega molto tempo blocca tutti. Ma i comandi
IRC sono O(n) sulla dimensione del server, non su operazioni di I/O.

27. PERCHE' LE ECCEZIONI E NON I CODICI DI RITORNO?
------------------------------------------------------
Ogni comando puo' fallire in molti punti (validazione parametri, verifica
permessi, ricerca canale, ricerca utente). Con codici di ritorno si avrebbe:

  if ((err = checkParams(msg)) != OK) return sendError(client, err);
  if ((err = findChannel(name)) != OK) return sendError(client, err);
  ...

Con le eccezioni il flusso normale e' lineare:
  Channel& chan = mustFindChannel(name);  // lancia se non esiste
  mustBeInChannel(client, chan);           // lancia se non e' membro
  mustBeOperator(client, chan);            // lancia se non e' op

Ogni funzione fa UNA cosa; gli errori si propagano automaticamente.
Il codice che gestisce il caso felice e' separato dal codice di errore.
